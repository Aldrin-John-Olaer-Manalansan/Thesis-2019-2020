Maximum of 32bytes(32 characters) per packet can be retrieved at a time

NRF Transmission Header Message(NRFTHM) format (1byte=8bits)
  b7<<<<b0
0bnnNNmmmm

	0b				  nn		  NN			mmmm
	 ^				  ^			  ^			  	  ^
binary format		T.D.T.		R.D.T.		command switch

>>>>>>>>>(nn) messenger device type(nag iindicate kung anong device ang nag transmit ng message nato , repeater and any device compatible)
* 0b00 - 0 - smart socket
* 0b01 - 1 - cctv
* 0b10 - 2 - hub
* 0b11 - 3 - repeater - information is requested to be REPEATED/Rebroadcast(receiver device will act as repeater), ginagamit pag di abot ng master/R-Pi or ng slave/device yung destination device.

>>>>>>>>>(NN) recepient device type(nag iindicate kung anong device ang dapat makareceive ng message nato)
* 0b00 - 0 - smart socket
* 0b01 - 1 - cctv
* 0b10 - 2 - hub
* 0b11 - 3 - all device type

>>>>>>>>>message switches(16 possible commands)
~~~~UNIVERSALLY/Any Device Related~~~~
* 0b0000 - 0  - information is a REQUEST(eg. esp32 requesting the wi-fi info,a device is requesting the "public channel info", r-pi is requesting a device power consumption or ac line state, etc...)
* 0b0001 - 1  - information is the "public channel"(used for registering all NRF in one channel/room)

~~~~Smart Socket Device Related~~~~
* 0b0010 - 2  - information is the "smart socket's Power Consumption"(majored for current sensor)
* 0b0011 - 3  - information is the "smart socket's AC Line State"(majored for the Relay)

~~~~IP Camera Related~~~~
* 0b0100 - 4  - information is the "Wi-Fi" SSID and Password
* 0b0101 - 5  - information is the "IP-Address" of the IP-Camera Device

nrf transmission scheme
- first 3 bytes is a redundant NRFTHM(see above), where every byte is the same NRFTHM
- purpose for error correction(redundant majority is the REAL Information signal)

- 14th byte is the "packet information" with format in binary "0bCCCCcccc"
- CCCC - total number of packets the message occupied, eg. 0000 means only one packet was used, 0011 means 4 packets was used - max of 16 packets an information can use
- cccc - received packet number - 0000 is the first packet and so on... Since NRF24 reception can only receive one packet(32bytes) at a time, this can tells the current packet number. and then merge all packets from 0th up to 15th packet as one information message
- for wifi credentials, CCCC is the number of packets the password uses, while cccc is the number of packets the ssid uses, when that happens, the packet info will become like this:
0bAAAAAEEEEEcccccS - 0b	   AAAAA				EEEEE				ccccc					  S
							 ^					  ^					  ^						  ^
					passwordpacketcount		ssidpacketcount		current packet		for ssid(0),password(1)
- next is the information(28 bytes/characters max) - do not worry since dito sa project natin gumawa ako ng protocol na multi-packet transmission, kaya ng system ntin na magsend ng 18 bytes per packet x 16 packets = 288 bytes or characters

data[0]--------->---------data[31]
0BzAARRRRRTTTTTPMMMMMMMMMMMMMMMMMM

   0B				z				   RRRRR					TTTTT				   AA					P							NMMMMMMMMMMMMMMMMM
	^				^			    	 ^						  ^					   ^					^									^
byte format		N.R.F.T.M.		Receiver Serial-Key		Transmitter Serial-Key		Trashbin		packet information		Information(18 bytes or characters max)

Note: the 'N' can be either an information character or a special packet information:
for Wifi Credentials, the packetinfo becomes like this:
0B PN = 0bpppppsssssCCCCCc

	0b 							ppppp 					  		  	    sssss 							CCCCC 									  c
	^							  ^										  ^						  		  ^										  ^
bit format		Password packets count(0=no password) 		SSID packets count(0=no ssid) 		current packet received 		packet was for (0 - SSID, 1 - password)

eg. info of 5 packets
packetp[0]=0B	96	77	66	36	55	45	76	99	87	32	100	120	172	32	125	54	55	< good message
packetp[1]=0B	55	45	44	64	76	255	43	78	55	45	64	33	76	108	54	111	32	< has a HOLE(0B 255) , the receiver will detect this packet as a "lost" packet
packetp[2]=0B	43	33	43	58	78	99	56	45	44	76	65	76	45	126	123	32	42	< good message
packetp[3]=0B	55	45	44	64	76	122	43	78	55	45	64	33	76	108	54	111	32	< good message
packetp[4]=0B	96	77	66	36	55	45	76	99	87	32	0	0	0	0	0	0	0	< has a TERMINATOR(0B 0) , the receiver will detect this as the end of the message

when a HOLE was detected by the receiver, it will send a "get lost packet" request()
